# Frozen Qt-curated notebooks — the "GUI marketing path."
#
# These notebooks carry hand-curated screenshots of the desktop
# ``MetalGUI`` Qt window (full window grabs: title bar, toolbar, dock
# panels, log dock, matplotlib canvas). The published docs site relies
# on them to show what the desktop experience looks like.
#
# CI policy
# ---------
#
# These notebooks are still executed on every PR by ``_dev/rerun_auto.py``
# (and the ``tests-lite`` CI job), BUT with ``--check-only`` semantics:
#
#   - Execution writes outputs to ``/tmp/<basename>.executed.ipynb``,
#     NOT back into the source file.
#   - Pass/fail is gated. The hand-curated outputs in git remain
#     untouched.
#
# That gives us the safety net (CI tells us if a notebook stopped
# running cleanly) without clobbering the rich Qt screenshots with the
# headless ``MetalGUIHeadless`` matplotlib renders that CI produces.
#
# Maintainer workflow
# -------------------
#
# When ``MetalGUI`` changes meaningfully (toolbar layout, theme,
# canvas styling) and the committed screenshots no longer reflect
# current behaviour:
#
#   1. Open the affected notebook locally in JupyterLab with
#      ``pip install "quantum-metal[gui]"`` active.
#   2. ``Restart Kernel and Run All`` — the real desktop ``MetalGUI``
#      paints into a real Qt window; ``gui.screenshot()`` grabs it.
#   3. Visually inspect; the maintainer-arranged dock layout produces
#      noticeably nicer screenshots than a default Qt session.
#   4. Sync: ``uv run python _dev/sync_two_folders.py --write``.
#   5. Commit.
#
# Future option (Option E in the PR thread)
# -----------------------------------------
#
# A manually-triggered ``workflow_dispatch`` CI job using ``Xvfb`` +
# ``quantum-metal[gui]`` can auto-refresh these notebooks in CI. The
# trade-off is that Qt session dock geometry defaults to an uglier
# layout than a hand-arranged maintainer session — auto-refreshed
# screenshots are functional but visually inferior. Deferred to a
# follow-up.
